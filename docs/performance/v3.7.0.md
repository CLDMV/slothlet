# Performance Benchmark — v3.7.0

> Adds **read-level permission gating** — property _reads_ of terminal data values are now permission-checked, not just function calls, and it is **on by default** when a `permissions` block is configured (`permissions.readGating: false` opts out). This benchmark runs with permissions **off** (the default), so the read gate short-circuits and startup/call numbers are unaffected; a small per-read cost applies only when permissions are configured with read gating on. Benchmark date: June 2026 — re-benchmarked from the `v3.7.0` tag on Node.js 24 with `--conditions=slothlet-dev`; numbers reflect this version's code on current hardware, not its original release environment.

## Summary

| Metric                                | Eager   | Lazy     | Winner | Delta          |
| ------------------------------------- | ------- | -------- | ------ | -------------- |
| **Startup** (10-iter avg)             | 24.52ms | 22.27ms  | Lazy   | ~2.3ms (noise) |
| **Function Calls** (100-iter avg)     | 5.65μs  | 5.49μs   | Lazy   | ~0.2μs (noise) |
| **Materialization** (lazy first call) | —       | 471.77μs | —      | one-time       |

## Notes

- Read gating only fires when permissions are configured; with no `permissions` block (this benchmark) `isEnabled()` short-circuits and reads are free. When enabled, a data-value read is checked against the rule set exactly like a call.
- Micro-benchmark caveat: run-to-run variance is high (startup spreads tens of ms; calls a few μs), so eager-vs-lazy deltas within a few ms / sub-μs are measurement noise, not real differences.
- Re-benchmarked from the release tag on current hardware/Node.js 24; original release-time numbers are not reproduced here.
