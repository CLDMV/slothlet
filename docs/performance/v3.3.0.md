**Conclusion:** Enabling permissions with `defaultPolicy: "allow"` and no rules adds zero measurable overhead. Even when enabled, the cost is within measurement noise because the benchmark modules make direct calls without rule evaluation (no caller context in the benchmark).

## Notes

- The permission system is off by default. When no `permissions` config is provided, `PermissionManager.isEnabled()` returns `false` and the `applyTrap` check short-circuits with zero work.
- The ~5ms increase in eager startup vs v3.2.3 is likely due to the additional handler registration (PermissionManager) and the heavier import graph. This is within normal run-to-run variance.
- Call-time differences between "off" and "on" are within noise (¬±2Œºs). The benchmark doesn't trigger rule evaluation since calls are made from the test harness (no caller wrapper context).
