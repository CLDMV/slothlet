# Performance Benchmark — v3.6.0

> Adds caller-identity controls for callbacks — `self.slothlet.lockCaller(fn)` pins the registering module's identity onto a callback, and `self.slothlet.bind(fn)` re-exports Node's `AsyncResource.bind`. These affect how `self.*` calls inside callbacks are attributed, not the hot call/startup path. Benchmark date: June 2026 — re-benchmarked from the `v3.6.0` tag on Node.js 24 with `--conditions=slothlet-dev`; numbers reflect this version's code on current hardware, not its original release environment.

## Summary

| Metric | Eager | Lazy | Winner | Delta |
|--------|-------|------|--------|-------|
| **Startup** (10-iter avg) | 21.83ms | 21.54ms | Lazy | ~0.3ms (noise) |
| **Function Calls** (100-iter avg) | 4.84μs | 4.86μs | Eager | ~0.0μs (tie) |
| **Materialization** (lazy first call) | — | 343.48μs | — | one-time |

## Raw Numbers

```text
Eager startup:  21.83ms
Lazy startup:   21.54ms
Eager calls:     4.84μs
Lazy calls:      4.86μs
Lazy first call (materialization): 343.48μs
```

## Notes

- `lockCaller`/`bind` only matter for callbacks that fire outside the registering module's async context; they add no cost to ordinary `self.*` calls — runtime perf tracks v3.5.0.
- Micro-benchmark caveat: run-to-run variance is high (startup spreads tens of ms; calls a few μs), so eager-vs-lazy deltas within a few ms / sub-μs are measurement noise, not real differences.
- Re-benchmarked from the release tag on current hardware/Node.js 24; original release-time numbers are not reproduced here.
