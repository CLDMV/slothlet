# Performance Benchmark — v3.4.0

> Extends the Permission System with **context-conditional rules** (an optional `condition` field evaluated against the per-request ALS context). Conditional rules are intentionally **never cached** — the manager skips the resolved-cache write when `hasConditionalRules` — so a rule set that uses conditions pays a per-evaluation cost; rules without `condition` are unchanged and still cached. Permissions remain **off by default** (zero overhead when unconfigured). Benchmark date: June 2026 — re-benchmarked from the `v3.4.0` tag on Node.js 24 with `--conditions=slothlet-dev`; numbers reflect this version's code on current hardware, not its original release environment.

## Summary

## Notes

- Perf-relevant change: conditional permission rules. A rule carrying a `condition` re-evaluates each call (not cached); unconditional rules keep their cached behavior. The default-off permission system adds no overhead in this benchmark.
- Micro-benchmark caveat: run-to-run variance is high (startup spreads tens of ms; calls a few μs), so eager-vs-lazy deltas within a few ms / sub-μs are measurement noise, not real differences.
- Re-benchmarked from the release tag on current hardware/Node.js 24; original release-time numbers are not reproduced here.
