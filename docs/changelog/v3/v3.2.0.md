# Slothlet v3.2.0 Changelog

**Release Date**: April 3, 2026
**Release Type**: Minor
**Branch**: `release/3.2.0`

---

## Overview

Version 3.2.0 introduces **API Path Versioning** — the ability to register the same logical API path (e.g. `auth`) multiple times under different version tags (e.g. `v1`, `v2`). A configurable discriminator automatically routes each caller to the correct version at dispatch time. Both the direct versioned namespaces (`api.v1.auth`, `api.v2.auth`) and the logical dispatcher path (`api.auth`) are live simultaneously.

This release also fixes a shutdown race in lazy mode where in-flight ESM `import()` calls could cause unhandled rejections when a worker process exited before those promises resolved.

No breaking changes. All v3.1.x configuration and API usage is fully compatible.

## 🚀 New Features

### API Path Versioning

Register the same logical API path under multiple version tags, and let Slothlet route calls to the right version automatically based on the caller's metadata.

#### Registering versioned paths

Pass a `versionConfig` object as the 4th argument to `api.slothlet.api.add`:

```javascript
const api = await slothlet({
	dir: "./api/base",
	versionDispatcher: "version" // read caller's versionMetadata.version
});

// Mount v1 — mark as the explicit default fallback
await api.slothlet.api.add("auth", "./api/v1/auth", {}, {
	version: "v1",
	default: true,
	metadata: { stable: true, deprecated: false }
});

// Mount v2
await api.slothlet.api.add("auth", "./api/v2/auth", {}, {
	version: "v2",
	metadata: { stable: true }
});
```

After both registrations:

- `api.v1.auth` → real v1 module (always available, bypasses dispatcher)
- `api.v2.auth` → real v2 module (always available, bypasses dispatcher)
- `api.auth` → dispatcher proxy (routes based on caller's version metadata)

#### Transparent dispatch

Callers use the logical path (`api.auth`) without knowing which version runs. The dispatcher resolves the version at property-access time:

```javascript
// Caller registered with { version: "v2" } → routes to v2
const result = api.auth.login("alice", "pass", "mfa-token");

// Direct versioned access bypasses the dispatcher entirely
const v1result = api.v1.auth.login("alice", "pass");
const v2result = api.v2.auth.login("alice", "pass", "mfa-token");
```

Different versions may have completely different argument signatures — the dispatcher intercepts **property access only** and never modifies arguments.

#### Discriminator configuration

Control how the version is selected at dispatch time:

```javascript
// String discriminator (default) — reads the named key from caller's version metadata
const api = await slothlet({ dir: "./api", versionDispatcher: "version" });

// Function discriminator — full custom routing logic
const api = await slothlet({
	dir: "./api",
	versionDispatcher: (allVersions, caller) => {
		// allVersions = { v1: { version, default, metadata, versionMetadata }, v2: { ... } }
		// caller      = { version, default, metadata, versionMetadata }
		if (caller.metadata.role === "admin") return "v2";
		return caller.version ?? null; // null falls through to the default version
	}
});

// Not set — behaves identically to versionDispatcher: "version"
const api = await slothlet({ dir: "./api" });
```

#### Default version resolution

When the discriminator returns nothing resolvable, Slothlet selects a default automatically:

1. An explicit `default: true` on a registered version wins.
2. Otherwise the **highest version** wins using semver-style comparison (leading `v`/`V` prefix and pre-release/build suffixes are stripped for comparison; original tag is preserved as the return value).

```text
Tags: ["v1", "v3", "v8", "v2"]  →  default: "v8"
Tags: ["v1", "2.0.0-alpha", "2.0.0", "v1.5"]  →  default: "2.0.0"
```

#### Inline version override

Force a specific version for one call using the ALS context symbol — useful for tests or admin overrides:

```javascript
import { context } from "@cldmv/slothlet/runtime";

context[Symbol.for("slothlet.versioning.force")] = "v1";
const result = api.auth.login("alice", "pass"); // always v1 regardless of discriminator
```

### `api.slothlet.versioning` — Runtime Version Management

A new `versioning` namespace is available on every Slothlet instance:

#### `api.slothlet.versioning.list(logicalPath)`

List all registered versions and the current default for a logical path.

```javascript
const info = api.slothlet.versioning.list("auth");
// {
//   versions: {
//     v1: { moduleID, versionTag, versionedPath, isDefault, versionMeta, registeredAt },
//     v2: { moduleID, versionTag, versionedPath, isDefault, versionMeta, registeredAt }
//   },
//   default: "v2"
// }
```

#### `api.slothlet.versioning.setDefault(logicalPath, versionTag)`

Override the default version at runtime without re-registering.

```javascript
api.slothlet.versioning.setDefault("auth", "v1");
```

#### `api.slothlet.versioning.unregister(logicalPath, versionTag)`

Remove a version. The dispatcher is updated if other versions remain; if all versions are removed the dispatcher is torn down and the logical path is removed from the API tree.

```javascript
const wasRemoved = await api.slothlet.versioning.unregister("auth", "v2");
// api.v2.auth is gone; api.auth now routes to v1 only
```

#### `api.slothlet.versioning.getVersionMetadata(logicalPath, versionTag)`

Retrieve the VersionManager-only metadata stored at registration time via `versionConfig.metadata`.

```javascript
const meta = api.slothlet.versioning.getVersionMetadata("auth", "v1");
// { version: "v1", logicalPath: "auth", stable: true, deprecated: false }
```

#### `api.slothlet.versioning.setVersionMetadata(logicalPath, versionTag, patch)`

Patch (merge) the VersionManager-only metadata for a registered version at runtime. The injected `version` and `logicalPath` keys always win over fields in `patch`.

```javascript
api.slothlet.versioning.setVersionMetadata("auth", "v1", { stable: true, deprecated: false });
// Immediately visible to the discriminator via allVersions.v1.versionMetadata
```

### `versionConfig.metadata` — Separate Version Metadata

`options.metadata` (regular Metadata system) and `versionConfig.metadata` (VersionManager) are two distinct stores that are never merged:

| System | Set via | Read via |
|---|---|---|
| Regular Metadata | `options.metadata` in `api.add` or `metadata.setForVersion()` at runtime | `module.__metadata` / `metadata.caller()` / `metadata.getForVersion()` |
| Version Metadata | `versionConfig.metadata` in `api.add` or `versioning.setVersionMetadata()` at runtime | `versioning.getVersionMetadata(logicalPath, versionTag)` / discriminator args |

The function discriminator receives both in separate named fields (`metadata` and `versionMetadata`) on each entry in `allVersions` and on `caller`.

Two convenience methods were added to `api.slothlet.metadata` for working with regular metadata on versioned modules without needing to construct the versioned path manually:

```javascript
// Set regular metadata on the v1 registration of "auth"
api.slothlet.metadata.setForVersion("auth", "v1", "stable", true);
api.slothlet.metadata.setForVersion("auth", "v1", { stable: true, region: "us" });

// Read back the user-supplied path metadata (no system fields)
const meta = api.slothlet.metadata.getForVersion("auth", "v1");
// { stable: true, region: "us" }

// Full metadata including system fields — access the namespace directly
const full = api.v1.auth.__metadata;
// { stable: true, region: "us", apiPath: "v1.auth", filePath: "...", moduleID: "..." }
```

The `__metadata` accessor on the logical dispatcher path (`api.auth.__metadata`) uses the same discriminator as a regular property access — returning the resolved version's metadata, not a merged view of all versions.

## 🐛 Bug Fixes

### Shutdown race in lazy mode

**Fix**: `_drainInFlightLoads()` is now called at the top of `shutdown()` before any teardown runs. It walks the API tree depth-first (cycle-safe, capped at 15 levels, skips version dispatcher proxies) and collects every pending `materializationPromise`. `Promise.allSettled()` waits for all in-flight ESM `import()` calls to settle before proceeding.

**Root cause**: In lazy mode, `api.slothlet.api.add()` and first property accesses trigger background ESM dynamic `import()` calls. When `api.shutdown()` was called immediately after (e.g. in an `afterEach`), the worker process could exit before those promises resolved, producing:

```text
SlothletError: [MODULE_IMPORT_FAILED] ... [vitest-worker]: Closing rpc while "fetch" was pending.
```

These surfaced as unhandled rejections in the full concurrent coverage run, causing exit code 1 even though all tests passed.

## ⚙️ Configuration Reference

### New config key: `versionDispatcher`

| Key | Type | Default | Description |
|---|---|---|---|
| `versionDispatcher` | `string \| function \| undefined` | `undefined` | Controls version routing. A string is the metadata key to look up on the caller; a function receives `(allVersions, caller)` and returns a tag or `null`; `undefined` behaves like `"version"`. |

### Updated `api.slothlet.api.add` signature

```javascript
api.slothlet.api.add(apiPath, folderPath, options, versionConfig)
```

`versionConfig` is an optional 4th argument:

| Field | Type | Required | Description |
|---|---|---|---|
| `version` | `string` | **Yes** | The version tag (e.g. `"v1"`, `"2.3.0"`) |
| `default` | `boolean` | No | Mark as the explicit default fallback |
| `metadata` | `object` | No | Version metadata stored in VersionManager only |

## Migration from v3.1.x

No breaking changes. All existing configuration and API surface is fully compatible.

The new `versionDispatcher` config key is optional and has no effect if not set. The new 4th argument to `api.slothlet.api.add` is optional and the existing 3-argument form is unchanged.
