# Slothlet v3.0.1 Changelog

**Release Date**: March 17, 2026
**Release Type**: Patch
**Branch**: `master`

---

## Overview

Version 3.0.1 is a patch release addressing three independent correctness issues found after the v3.0.0 launch: a resolver bug that mis-classified user entry points as internal Slothlet frames, two CI test-environment hardening issues, and a fragile `build:dist` script that crashed when `src/` had already been removed.

No public API or configuration changes. All existing v3.0.0 code is fully compatible without modification.

---

## 🐛 Bug Fixes

### Resolver: Caller `index.mjs` No Longer Treated as Internal

**Problem**: `#isSlothletInternal()` in `resolve-from-caller.mjs` used a bare `path.basename()` check to identify the Slothlet package entry points (`index.mjs` / `index.cjs`). Any file named `index.mjs` in a user's project — including their own application entry point — matched the same condition and was incorrectly treated as an internal Slothlet frame. This caused `resolveFromCaller()` to skip the wrong stack frame and return an incorrect caller path.

**Fix**: Added `SLOTHLET_PKG_ROOT` constant (`path.normalize(path.resolve(__dirname, "../../.."))`) and scoped the basename check so it only matches files whose immediate parent directory equals the package root. Files named `index.mjs` outside the Slothlet package root are no longer affected.

**Regression tests added**: Two new tests in `resolve-from-caller-paths.test.vitest.mjs` — one asserting the resolved path equals the fixture's own directory (not `process.cwd()`), one asserting it contains `"fixtures"`. Full suite of 21 tests passes.

**Files changed**:
- `src/lib/helpers/resolve-from-caller.mjs` — add `SLOTHLET_PKG_ROOT`, guard `#isSlothletInternal()` basenames
- `tests/vitests/suites/helpers/fixtures/index.mjs` — new regression fixture
- `tests/vitests/suites/helpers/resolve-from-caller-paths.test.vitest.mjs` — two new regression tests

---

### CI: `test-conditional.mjs` Slothlet-Dev Condition Stripping

**Problem**: `buildTestNodeEnv()` used exact token equality (`token !== "--conditions=slothlet-dev"`) to strip the `slothlet-dev` export condition from `NODE_OPTIONS` before running the post-build Node.js test. This exact match silently failed for compound conditions (`--conditions=slothlet-dev,other`) and space-separated form (`--conditions slothlet-dev`), leaving `slothlet-dev` active in the post-build environment and producing `Cannot find module '.../src/slothlet.mjs'` errors after `ci:cleanup-src` removed `src/`.

**Fix**: Replaced the single-token comparison with a full token-aware parser that handles both `--conditions=VALUE` and `--conditions VALUE` forms, stripping any comma-delimited component equal to `slothlet-dev`. Also replaced the `|| undefined` fallback (serialized as the literal string `"undefined"` by Node 16's `child_process.spawn`) with `delete env.NODE_OPTIONS` when the value becomes empty.

**Files changed**:
- `tests/test-conditional.mjs` — rewrite `buildTestNodeEnv()` stripping logic

---

### CI: `run-all-tests.mjs` Respawn Race Condition

**Problem**: `run-all-tests.mjs` called `if (ensureDevEnvFlags()) process.exit()` in the parent process to handle the "respawn with `slothlet-dev`" case. The parent exited immediately after spawning the child, discarding the child's real exit code before it could propagate. Test failures inside the child process were silently swallowed; the parent always exited `0`.

**Fix**: Restructured control flow to `if (!ensureDevEnvFlags()) { runAllTests(); }`. The parent process now stays alive, child stdio streams directly to the terminal, and `child.on("exit", code => process.exit(code))` correctly propagates the real exit code.

**Files changed**:
- `tests/run-all-tests.mjs` — invert `ensureDevEnvFlags()` guard, propagate child exit code

## 🛠️ Build

### Resilient `build:dist` Script

**Problem**: The `build:dist` npm script used `shx cp -r src/* dist/`, which throws a hard error when `src/` does not exist. Because `ci:cleanup-src` runs at the end of `build:ci`, any subsequent re-run of `build:dist` in CI (e.g., a retry) would crash immediately without a useful error message.

**Fix**: Replaced the `shx cp` invocation with `node tools/build/build-dist.mjs`. The new script uses `fs.cpSync` (Node 16.7+) with an `existsSync` guard: if `src/` is absent it exits cleanly with a warning rather than throwing. Normal builds are unaffected.

**Files changed**:
- `tools/build/build-dist.mjs` — new resilient copy script
- `package.json` — `build:dist` updated to `node tools/build/build-dist.mjs`
