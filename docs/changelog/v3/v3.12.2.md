# Slothlet v3.12.2 Changelog

**Release Date**: July 2026
**Release Type**: Patch
**Branch**: `release/3.12.2`

---

## Overview

Version 3.12.2 is a maintenance patch. Its library-facing fixes are both in type generation: a generated `.d.ts` now reflects the real JSDoc `@param` / `@returns` types of JavaScript leaves instead of emitting `any` for all of them (#213), and it compiles for TypeScript leaves that reference a local `interface` / `type` / `enum` — previously an undefined-name error (#219). The release also adds a testing guide covering the `server.deps.inline` requirement for measuring coverage of slothlet-composed leaves in a **consumer** project (#217), stabilizes the `tsc`-fork type-generation tests under coverage instrumentation (#216), and scopes two CI workflows to least privilege (#214, #210). No breaking changes.

---

## 🐛 Bug Fixes

### Generated `.d.ts` emitted `any` for JSDoc-typed JavaScript leaves (#213)

`slothlet typegen` (and strict-mode `.d.ts` generation) parsed each backing file with a bare `ts.createSourceFile` and read the type positions as raw text, so a JavaScript leaf's JSDoc annotations were dropped — every parameter and return came out `any`. A TypeScript (`.ts`) leaf typed correctly because its annotations are real syntax; a JSDoc-annotated `.mjs` / `.js` leaf did not.

Type extraction now builds a single TypeScript **Program** over the backing files and resolves each exported function's signature through the type **checker** (`signatureToString`). The checker reads JSDoc the way TypeScript itself does, so `@param {string}` / `@returns {Promise<User>}`, object shapes from a `@param {object} opts` tag plus its dotted `opts.id` sub-tags, optional parameters, unions, and generics all reflect into the generated signatures as proper, compiling TypeScript — no raw JSDoc leaks into the output. A file that can't be resolved contributes no signatures rather than failing the whole run. The Program runs with `checkJs: false`: it _reads_ the JSDoc types, it does not type-check the sources.

The generator's acceptance test now compiles the emitted declaration with `tsc` under a full semantic check (`skipLibCheck: false`, filtered to the generated file), so "the output is valid, usable TypeScript" is verified rather than assumed — the previous string-match test would have accepted a non-compiling declaration.

### Generated `.d.ts` failed to compile for `.ts` leaves using local named types (#219)

The TypeScript-side companion to #213. When a `.ts` leaf referenced a **local** (non-exported) named type — an `interface`, `type` alias, or `enum` declared in the same file — in a parameter or return, the checker printed that type by name, but the generated declaration never defined it, so `tsc` reported _"Cannot find name 'Point'"_ and the `.d.ts` did not compile. Primitive, inline-object, and exported types were unaffected; the gap was specific to local named types, and it predated #213.

The generator now runs TypeScript's **own declaration emitter** over the program and reads the referenced `interface` / `type` / `enum` declarations out of the emitted `.d.ts` — so each is written in exactly the ambient form `tsc` produces (`declare enum`, filled enum initializers), rather than a hand-massaged source paste. A lexical name-closure over the finite set of local type names selects the transitively-referenced ones (a type used only inside another type is pulled in; one nothing references is left out) — deliberately not a type-graph walk, which recursive generics would instantiate without bound. JavaScript / JSDoc leaves are unaffected: with no local named types, nothing extra is emitted. The acceptance test compiles the result, so a `.ts` API's generated declaration is now verified to build.

## 🧪 Tests

### `tsc`-fork type-generation tests get a 60s budget under coverage (#216)

The TypeScript strict-mode suite boots slothlet in a mode that forks a `tsc` type-generation pass. Under coverage instrumentation that fork runs materially slower (observed 12–35s), and the one case that performs the boot in a `beforeEach` hook exceeded vitest's default 10s hook budget — flaking the coverage run while passing normally. The file now sets a 60s test/hook budget for these `tsc`-fork boots (the same budget the operation already has everywhere else in the file), and the heaviest suite is scheduled on the runner's solo/serial lane so it isn't contending with parallel workers for resources. A test-infrastructure fix; no library behavior changes.

## 📚 Documentation

### NEW: measuring coverage in a consumer project (#217)

New guide [docs/TESTING.md](../../TESTING.md) documents a coverage-attribution artifact that surfaces when a **consumer** project (one that installs `@cldmv/slothlet` from npm) measures coverage of its own API leaves. A leaf reached only through slothlet composition reads as near-zero covered even though it executes and its output is asserted: slothlet is an externalized `node_modules` dependency, so its per-instance `import()` of each leaf is a native import that never enters vitest's module runner, and `@vitest/coverage-v8` only attributes modules in that graph. The fix is a single config line — add `@cldmv/slothlet` to `server.deps.inline` — with a measured before/after and the fidelity trade-off spelled out. Linked from the README documentation index and REFERENCE.md.

### TYPESCRIPT.md — how JavaScript and TypeScript leaves are typed

[docs/TYPESCRIPT.md](../../TYPESCRIPT.md) now states explicitly how both leaf kinds are typed: a JavaScript leaf's JSDoc `@param` / `@returns` are reflected (#213); a TypeScript leaf is typed from its annotations (JSDoc _type_ tags are ignored in `.ts` / `.mts`, as `tsc` ignores them) and any local `interface` / `type` / `enum` it references is emitted alongside the declaration (#219).

## 🔧 CI / Workflow

- **Scorecard permissions scoped to the `analyze` job (#214).** `scorecard.yml` declared its elevated permissions at the workflow level, granting them to the whole file; they are now granted on the single `analyze` job (`id-token: write`, `contents: read`, `actions: read`) instead. Workflow-level `security-events: write` is deliberately omitted while `publish_results: true`, since granting it makes scorecard-action's publish-verification step fail — the public OpenSSF badge is published via the reusable workflow, not a SARIF upload from this job.
- **Feature-PR flow labeling no longer mislabels non-feature branches (#210).** `feature-pr.yml` applied its feature-lane flow label to PRs from branches that are not feature branches; the label logic now scopes correctly so only feature branches get the lane label.

## Upgrade notes

- **No breaking changes.**
- **Regenerating types now produces better output.** If you commit a generated `.d.ts` (or a types package) and regenerate it under v3.12.2, expect a diff: JavaScript leaves that were previously `any` now carry their JSDoc `@param` / `@returns` types (#213), and a TypeScript leaf that references a local `interface` / `type` / `enum` now emits that type into the declaration so it compiles (#219, previously an undefined-name error). The new declaration is more precise and valid TypeScript — review and commit the improved types. No action is needed if you don't regenerate.
