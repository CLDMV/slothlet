# Slothlet v3.3.1 Changelog

**Release Date**: April 26, 2026
**Release Type**: Patch
**Branch**: `release/3.3.1`

---

## Overview

Version 3.3.1 is a bug-fix release that closes a silent failure when user code calls `new` on a Slothlet-proxied class export.

No breaking changes. All v3.3.0 configuration and API usage is fully compatible.

---

## 🐛 Bug Fixes

### `new` on proxied class exports now works correctly

**Before:** Calling `new api.someModule.MyClass(args)` silently produced an empty plain object `{}`. The real constructor body never ran, no own properties were set, and the prototype chain was missing (methods were `undefined`). No error was thrown because JavaScript's `[[Construct]]` fallback constructs the empty stub function that Slothlet uses as a proxy target.

**After:** `new api.someModule.MyClass(args)` returns a fully constructed instance with all own properties set and the complete prototype chain intact.

#### Root cause

The `Proxy` handler assembled in `UnifiedWrapper.createProxy()` had no `construct` trap. Without one, `new proxy(args)` falls back to `Reflect.construct(proxyTarget, args)` where `proxyTarget` is an empty `function(){}` stub — the constructor body of the real class never runs.

Additionally, the `newTarget` argument must be the real `impl` function (not the proxy). Passing the proxy as `newTarget` routes `newTarget.prototype` through the `getTrap`, which returns a child wrapper instead of the real class prototype, breaking the prototype chain even when the constructor body ran.

#### What changed

A `constructTrap` was added to the proxy handler in `src/lib/handlers/unified-wrapper.mjs`:

- **Eager mode / already-materialized:** delegates directly via `Reflect.construct(impl, args, impl)`.
- **Lazy mode, in-flight:** the trap returns a `Promise` so callers can `await new api.MyClass(args)` and receive the real instance once materialization completes.
- **Lazy mode, not yet materialized:** fires `_materialize()` then enters the in-flight path above.
- **Non-callable impl:** throws `INVALID_CONFIG_NOT_A_FUNCTION`, consistent with `applyTrap`.

The constructed instance is returned **unwrapped** — no re-proxying — so private class fields (stored in per-instance `WeakMap`s keyed on the raw instance) continue to work correctly.

#### Example

```javascript
// api/adapters/openai-backend.mjs
export class OpenAIBackendAdapter {
	#token;
	constructor(config) {
		this.model = config.model;
		this.#token = config.token;
	}
	listModels() { return ["gpt-4o"]; }
}

// Consumer (eager mode)
const adapter = new api.adapters.openaiBackend.OpenAIBackendAdapter({ model: "gpt-4o", token: "sk-..." });
console.log(adapter.model);         // "gpt-4o"   ← was {} before
console.log(adapter.listModels());  // ["gpt-4o"] ← was TypeError before

// Consumer (lazy mode — await the constructor call)
const adapter = await new api.adapters.openaiBackend.OpenAIBackendAdapter({ model: "gpt-4o", token: "sk-..." });
```

## 🧪 Tests

- Added `api_tests/api_test/classes.mjs` — fixture with `Counter` (own properties + prototype methods) and `Secrets` (private field via `#token`).
- Added `tests/vitests/suites/handlers/unified-wrapper-construct-trap.test.vitest.mjs` — 40 tests across all 8 matrix configurations (eager × 4, lazy × 4) covering:
  - Constructor body execution (own property set)
  - Prototype chain preservation (methods callable)
  - Instance identity (`not.toEqual({})`)
  - Private class field correctness (no re-proxy)
  - Prototype consistency across multiple `new` calls
