# Slothlet v3.2.2 Changelog

**Release Date**: April 12, 2026
**Release Type**: Patch
**Branch**: `release/3.2.2`

---

## Overview

Version 3.2.2 is a patch release that adds a missing `set` trap to the version dispatcher proxy introduced in v3.2.0. Without the trap, plain property assignment of the form `self.versionedPath.prop = value` would throw a `TypeError: Cannot redefine property` in strict mode — even on the very first write. This made it impossible for any versioned module to store runtime state (e.g. a database client, a connected socket) on a versioned API path via the dispatcher.

No public API or configuration changes. All v3.2.0 and v3.2.1 usage is fully compatible.

---

## 🐛 Bug Fixes

### Dispatcher Proxy: Missing `set` Trap

**Problem**: The version dispatcher proxy had no `set` trap. Without it, JavaScript's engine uses the default `[[Set]]` path (`OrdinarySetWithOwnDescriptor`), which consults `[[GetOwnProperty]]` on the receiver (the dispatcher proxy itself) before writing. The dispatcher's `getOwnPropertyDescriptor` trap returns `{ writable: false }` for every unknown property — a deliberate choice to avoid prematurely invoking version resolution during property enumeration. As a result, the default `[[Set]]` sees `writable: false` and throws `TypeError: Cannot redefine property: <prop>` in strict mode before the value is ever written.

**Impact**: Any module that stored runtime state on a versioned API path through the dispatcher — e.g.:

```js
// inside a versioned module using self
self.cache.redisClient = client;
self.db.connection = pool;
```

would crash with `TypeError: Cannot redefine property`, leaving the module in a broken half-initialized state. The direct versioned path (`self.v1.cache.redisClient = client`) worked correctly because it hits `UnifiedWrapper` directly, bypassing the dispatcher.

**Fix**: Added a `set` trap to `createDispatcher()` that delegates via `Reflect.set(vw, prop, value, vw)` to the real versioned wrapper's own `setTrap`. `UnifiedWrapper.setTrap` performs the correct delete → `Object.defineProperty(configurable: true)` cycle, keeping properties perpetually reassignable.

The trap also guards against writes to keys where the `get` trap returns fixed values rather than delegating to the versioned wrapper (framework internals, stable accessors, well-known string props such as `toString`/`valueOf`/`toJSON`/`then`, and all symbols). Forwarding writes for those keys to the versioned wrapper would create invisible hidden state — the write appears to succeed but is never observable through the dispatcher's `get` trap. Those writes are silently absorbed (`return true`) to keep `set`/`get` behaviour consistent.

**Keys absorbed (not forwarded to versioned wrapper)**:

- Framework internals: `____slothletInternal`, `_impl`, `__impl`, `__state`, `__invalid`
- Stable framework accessors: `__isVersionDispatcher`, `__mode`, `__apiPath`, `__slothletPath`, `__isCallable`, `__materializeOnCreate`, `__materialized`, `__inFlight`, `__displayName`, `__moduleID`, `_materialize`, `length`, `name`
- Structural / thenable props: `then`, `constructor`, `toString`, `valueOf`, `toJSON`
- All symbols (including `Symbol.toStringTag` — `UnifiedWrapper` computes it dynamically on reads, so forwarding writes would create unobservable hidden state)

**Keys forwarded to versioned wrapper** (get delegates to vw): `__metadata`, `__filePath`, `__sourceFolder`, `__type`, and all user-defined string properties.

## 🧪 Tests

### New Suite: `versioning-dispatcher-set.test.vitest.mjs`

Added 9 test cases (72 total across the full matrix) covering:

- First assignment through dispatcher does not throw
- Assigned value is readable back through the dispatcher
- Value is stored on the versioned wrapper, not the dispatcher target
- Repeated assignment (reassignment cycle) does not throw
- Multiple distinct properties can be assigned
- Assignment routes to the active default version when two versions exist
- Writes to framework accessor keys are silently absorbed without throwing and without changing what `get` returns
- Writes to symbol keys are silently absorbed and do not appear on the versioned wrapper
- Absorbed writes do not pollute the versioned wrapper
