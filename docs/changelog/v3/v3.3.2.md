## Summary

Follow-up patch to the v3.3.1 release. The `release.yml` and `publish.yml` workflows were missed when `ci.yml` was updated for the Node.js ≥ 20.19.0 engine requirement. This fix brings all three workflows into alignment and adds the `lts_only_matrix` flag that was introduced in `ci.yml` during the v3.3.1 cycle.

---

## Changes

### `release.yml`

- `min_node_version` default changed from `16.20.2` → `20.19.0`
- Added `lts_only_matrix` boolean input (default: `true`)
- Passes `lts_only_matrix: ${{ github.event.inputs.lts_only_matrix != 'false' }}` to the reusable workflow

### `publish.yml`

- `min_node_version` default changed from `16.20.2` → `20.19.0`
- Added `lts_only_matrix` boolean input (default: `true`)
- Passes `lts_only_matrix: ${{ github.event.inputs.lts_only_matrix != 'false' }}` to the reusable workflow

### `README.md`

- Updated minimum Node.js requirement from `v16.20.2` to `v20.19.0`
- Removed outdated note about Node.js 16.4–16.19 stack trace regression workaround

---

## Background

`lts_only_matrix` restricts the Node.js version matrix to even-numbered (LTS) majors (20, 22, 24, …). This is required because some transitive dependencies declare `engines` fields that exclude odd-numbered Node.js releases. Without this flag the release and publish workflow matrix tests would fail on Node 21 / 23 despite the package itself being fine.
