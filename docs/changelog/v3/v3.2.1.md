# Slothlet v3.2.1 Changelog

**Release Date**: April 11, 2026
**Release Type**: Patch
**Branch**: `release/3.2.1`

---

## Overview

Version 3.2.1 is a patch release that hardens the dispatcher proxy introduced in v3.2.0. Three correctness issues in the `defineProperty` trap were identified and fixed via Copilot code review, ensuring the trap correctly propagates failures and cannot leave the versioned wrapper in a partially-mutated state when proxy invariants cannot be satisfied.

A minor pre-commit tooling cleanup is also included.

No public API or configuration changes. All v3.2.0 usage is fully compatible.

---

## ЁЯРЫ Bug Fixes

### Dispatcher `defineProperty` Trap: `Object.defineProperty` тЖТ `Reflect.defineProperty`

**Problem**: The `defineProperty` trap forwarded property definitions to the resolved versioned wrapper using `Object.defineProperty(vw, prop, descriptor)`. If the versioned wrapper's own `defineProperty` trap returned `false` (e.g., to reject the operation), `Object.defineProperty` throws a `TypeError` instead of returning `false`. The dispatcher trap never gets to evaluate its own return path тАФ the exception propagates before it can.

This means callers using `Reflect.defineProperty(api.dispatcher, prop, descriptor)` and checking the boolean return value would receive an exception instead of a clean `false`, breaking any error-handling code that expected the non-throwing semantics of `Reflect.defineProperty`.

**Fix**: Replaced `Object.defineProperty(vw, prop, descriptor)` with `return Reflect.defineProperty(vw, prop, descriptor)`. The return value of `vw`'s trap now flows back to the caller correctly.

---

### Dispatcher `defineProperty` Trap: Shadow Preflight + Versioned Wrapper First

**Problem**: The original code mirrored the raw target `t` first, then wrote to the versioned wrapper `vw`. If `vw` rejected the definition (e.g., the property was already non-configurable on `vw` with an incompatible value), `t` was left with an orphaned non-configurable property. The `getOwnPropertyDescriptor` trap reports the `t` descriptor while the `get` trap returns the differing `vw` value тАФ V8's ┬з10.5.8 proxy invariant then throws a `TypeError` on the very next property read.

**Fix**: For `configurable: false` descriptors the trap now: (1) performs a shadow preflight тАФ cloning `t`'s current descriptor and extensibility onto a transient object so the engine's rules decide compatibility without mutating `t`; (2) writes to `vw` first; (3) only if `vw` accepts, mirrors onto `t` to satisfy V8's ┬з10.5.6 step 28 invariant. If either the preflight or `vw` rejects, the trap returns `false` and `t` is never touched.

---

### Dispatcher `defineProperty` Trap: Incompatible Redefine of Already-Sealed Property

**Problem**: The raw-target mirror for `configurable: false` descriptors was guarded by `if (!existing || existing.configurable !== false)`. This guard was added to avoid "Cannot redefine property" errors on second identical writes. However, it over-skipped: if a caller attempted to redefine an already-non-configurable property with a **different** (incompatible) value, the guard suppressed the raw-target validation entirely. The versioned wrapper was then mutated and `true` returned. V8's proxy invariant check then saw that `t` held a different descriptor from what the trap claimed, and threw a `TypeError` after the mutation had already been applied.

**Root cause**: The guard prevented `Reflect.defineProperty(t, ...)` from running the plain-object compatibility check (same-value semantics) that correctly distinguishes compatible redefinitions (same value тЖТ succeeds) from incompatible ones (different value тЖТ returns `false`).

**Fix**: Removed the `existing.configurable !== false` guard entirely. `Reflect.defineProperty(t, prop, descriptor)` is now called unconditionally for all `configurable: false` descriptors. Plain-object semantics handle both cases safely:
- Same-value redefine on already-sealed property тЖТ `Reflect.defineProperty` returns `true` тЖТ write proceeds
- Different-value redefine on already-sealed property тЖТ `Reflect.defineProperty` returns `false` тЖТ trap returns `false` without touching the versioned wrapper

A regression test was added to `versioning-dispatcher-internals.test.vitest.mjs` covering this path.

## ЁЯЫая╕П Tooling

### `precommit-validation.mjs`: Removed Duplicate `build:cleanup` Step

The explicit `"Clean Build Artifacts"` step in `validationSteps` ran `npm run build:cleanup` at the start of the pre-commit sequence. However, `npm run build:dev` (the next step) already starts with `build:cleanup` as its first sub-command. This caused `dist/` and `types/` to be deleted twice per commit with no benefit.

**Fix**: Removed the standalone `build:cleanup` step. The sequence is now:
`build:dev тЖТ debug тЖТ test:node тЖТ vitest`

The stale `"Clean Build Artifacts"` entry was also removed from the `stepCommands` failure-hint map, and the `@fileoverview` JSDoc was updated to reflect the new sequence.
